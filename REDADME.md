# FastAPI микросервис для торговых данных

Микросервисное приложение на FastAPI, которое использует данные из таблицы spimex_trading_results и отдает их в формате JSON с кэшированием через Redis.
___
## Запуск проекта

### 1. Клонировать репозиторий

```bash
git clone https://github.com/temashev/fastapi-section
cd fastapi-section
```
### 2. Настройка окружения
В корне проекта необходимо создать `.env` файл (по примеру `.env.example`)\
```bash
cp .env.example .env
```
### Переменные окружения
* `DB_NAME` - Имя базы данных PostgreSQL.
* `DB_USER` - Имя пользователя БД.
* `DB_PASS` - Пароль от БД.
* `DB_HOST` - Хост для подключения.
* `DB_PORT` - Порт БД (по умолчанию `5432`).
* `REDIS_HOST` - Имя хоста для Redis
* `REDIS_PORT` - Порт Redis (по умолчанию `6379`)

### 3. Установка зависимостей
```bash
python -m venv .venv
source .venv/Scripts/activate
pip install -r requirements.txt
```

### 4. Запуск сервера
```bash
uvicorn app.main:app --reload
```
Проект будет доступен по адресу `http://127.0.0.1:8000/`

Доступные ручки:
* `/api/v1/trading/dates/`
* `/api/v1/trading/dynamics/`
* `/api/v1/trading/results/`

### 5. API Документация
После запуска сервера интерактивная документация к эндпоинтам доступна по адресу:
* Swagger UI: `http://127.0.0.1:8000/docs/`

### 6. Запуск тестов
Для проверки работоспособности кода и логики приложения:
* Модульные тесты для бизнес-логики
    ```bash
    pytest tests/test_business.py
    ```
* Интеграционные тесты для API endpoints и Тесты кэширования с mock Redis
  ```bash
  pytest tests/test_api.py
  ```
* Тесты производительности (load testing)
  ```bash
  python tests/load_test.py
  ```

### Спецификация API Endpoints
#### 1. Список дат торгов
* GET `/api/v1/trading/dates/`
* Описание: Возвращает упорядоченный список последних уникальных дат, когда проводились торги.
* Параметры запроса:

    | Параметр  | Тип | Обязательный |                       Описание                        |
    |:---------:|:---:|:------------:|:-----------------------------------------------------:|
    | last_days | int |     Нет      | Количество последних торговых дней (по умолчанию: 30) |
___
#### 2. Динамика торгов за период
* GET `/api/v1/trading/dynamics/`
* Описание: Отдает агрегированную динамику торгов за указанный промежуток времени. Результаты кэшируются.
* Параметры запроса:

    |     Параметр      | Тип  | Обязательный |          Описание           |
    |:-----------------:|:----:|:------------:|:---------------------------:|
    |    start_date     | date |      Да      | Начало периода (ГГГГ-ММ-ДД) |
    |     end_date      | date |      Да      | Конец периода (ГГГГ-ММ-ДД)  |
    |      oil_id       | str  |     Нет      | Фильтр по ID нефти/продукта |
    | delivery_type_id  | str  |     Нет      |   Фильтр по типу поставки   |
    | delivery_basis_id | str  |     Нет      |  Фильтр по базису поставки  |
___

#### 3. Результаты торгов
* GET `/api/v1/trading/results/`
* Описание: Выводит детальный список результатов торгов по заданным фильтрам.
* Параметры запроса:

    |      Параметр      | Тип | Обязательный |                             Описание                              |
    |:------------------:|:---:|:------------:|:-----------------------------------------------------------------:|
    |       oil_id       | str |     Нет      |                         ID нефтепродукта                          |
    |  delivery_type_id  | str |     Нет      |                         ID типа поставки                          |
    | delivery_basis_id  | str |     Нет      |                        ID базиса поставки                         |
    |       limit        | int |     Нет      | Ограничение количества записей (по умолчанию `100`, макс. `1000`) |
___
## Система кэширования (Redis)

Для снижения нагрузки на PostgreSQL и ускорения ответов в проекте реализована кастомная логика кэширования с жесткой привязкой к расписанию обновления торговых данных.

### 1. Архитектура и правила кэширования
*  Кэширование применяется ко всем трем эндпоинтам API.
*  Данные сериализуются в формат **JSON** перед отправкой в Redis.
*  Уникальный ключ кэша формируется динамически на основе всех параметров входящего запроса (включая query-параметры фильтрации и лимиты).

### 2. Логика обработки запроса (Cache Flow)
1. При поступлении запроса приложение генерирует ключ и проверяет его наличие в Redis.
2. Если ключ найден, данные мгновенно отдаются из кэша в лог пишется: 
   `[CACHE HIT] Данные успешно получены из кэша.`
3. Если ключа нет, приложение делает запрос к PostgreSQL, сохраняет результат в Redis с рассчитанным TTL и отдает ответ. В лог пишется:
   `[CACHE MISS] Данные не закэшированы.`

### 3. Инвалидация и время жизни (TTL) по расписанию
Так как новые итоги торгов на бирже публикуются раз в день, в приложении реализована кастомная логика очистки:
* Время жизни каждого ключа рассчитывается индивидуально при записи - строго до 14:11 текущего дня.
*  Ежедневно в 14:11 срабатывает запланированная фоновая задача (`Background Task`), которая полностью очищает (инвалидирует) устаревший кэш, подготавливая систему к получению свежих данных.
